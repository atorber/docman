# PRD生成 - 交互式澄清提示

## 使用说明

本文件包含PRD生成过程中各种交互确认场景的提示模板。执行skill时，根据实际情况选择使用相应的提示。

---

## 场景1：功能优先级确认

```
# 功能优先级确认

根据需求描述，需要确认此需求的优先级分类（MoSCoW原则）：

**需求概述**：
{需求描述摘要}

请问此需求的优先级属于哪一类?

选项:
  - A. Must have (必须有)
    - 这是核心功能，如果没有，产品无法正常使用或满足基本需求
    - 阻塞性需求，必须在当前版本实现
  - B. Should have (应该有)
    - 这是重要功能，但不阻塞发布
    - 如果时间允许应该在当前版本实现
  - C. Could have (可以有)
    - 这是锦上添花的功能
    - 可以延迟到后续版本实现
  - D. Won't have (不会有)
    - 明确不在当前范围的功能
    - 需要明确说明不做的原因

请选择: [A/B/C/D]
```

```
# 竞品策略确认

已识别到以下竞品产品：
{竞品列表及链接}

对于竞品功能，您的策略是?

选项:
  - A. 借鉴竞品
    - 参考竞品的功能设计和用户体验
    - 实现类似的功能和交互方式

    - 突出产品的独特价值和优势
  - C. 超越竞品
    - 在竞品基础上进行优化
    - 提供更好的功能和体验
  - D. 不参考竞品
    - 完全自主创新
    - 不参考竞品的设计

请选择: [A/B/C/D]
```

## 使用建议

1. **渐进式确认**：不要一次性问太多问题，根据生成进度逐步确认
2. **提供上下文**：每个问题都提供足够的背景信息，帮助用户决策
3. **默认值友好**：为复杂问题提供合理的默认选项
4. **记录决策**：将所有用户决策记录到文件，便于追溯
5. **灵活调整**：根据用户反馈动态调整生成策略

## 输出格式

所有用户回答应按以下格式记录：

```markdown
## 交互记录

### 场景{编号}: {场景名称}

**时间**: {ISO8601时间戳}
**问题**: {问题内容}
**用户回答**: {用户选择或补充内容}
**决策影响**: {此决策对PRD生成的影响}
```

最终汇总到 `references/交互记录/{时间戳}_clarifications.md` 文件中。
