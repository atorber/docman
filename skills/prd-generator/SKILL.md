# PRD生成工具

本skill用于根据用户提供的初始需求、竞品分析等信息，自动生成完整的产品需求文档(PRD)。支持交互式确认机制，确保生成的PRD符合用户预期。

## 适用场景

- 新功能需求的PRD编写
- 功能优化的需求文档
- 系统重构的需求说明
- 缺陷修复的需求描述
- 需求文档的标准化整理

## 调用方式

用户通过prompt提供以下参数：

| 参数 | 格式 | 必填 | 说明 |
|------|------|------|------|
| 需求类型 | `新功能/功能优化/重构/修复` | ✅ | PRD类型 |
| 产品名称 | `xxx平台` | ✅ | 产品名称 |
| 需求标题 | `简短标题` | ✅ | 需求名称 |
| 初始PRD路径 | `prd/xxx.md` | ❌ | 已有的简单PRD(可选) |
| 需求描述 | `自然语言描述` | ✅ | 核心需求说明 |
| 用户画像 | `目标用户描述` | ❌ | 目标用户群体 |
| 竞品链接 | `URL列表` | ❌ | 竞品产品文档链接 |
| 参考文档 | `文件路径/URL` | ❌ | 相关参考材料 |
| 输出路径 | `prd/xxx/xxx.md` | ✅ | PRD输出路径 |
| 时间戳 | `YYYYMMDD_HHMMSS` | ✅ | 文件命名时间戳 |

**prompt示例**：
```
# PRD生成任务

## 输入信息
- 需求类型: 新功能
- 产品名称: AIHC智能计算平台
- 需求标题: 数据集管理功能
- 需求描述: 用户需要创建、管理、删除数据集，支持数据集的版本控制和权限管理
- 用户画像: 数据科学家、算法工程师
- 竞品链接: https://cloud.google.com/vertex-ai/datasets
- 输出路径: prd/数据集管理/数据集管理_PRD.md

## 输出配置
- 时间戳: 20260428_153000
```

## 执行流程

### 阶段一：信息收集与分析

**目标**：收集并整理生成PRD所需的所有信息

#### 1.1 读取现有材料

根据提供的路径读取以下内容：
- 初始PRD文档（如有）
- 竞品文档内容（使用web_fetch抓取）
- 参考文档（如有）

支持多种文档格式：
- Markdown (.md)：直接读取
- Word (.docx)：解析文本内容
- 文本文件 (.txt)：直接读取
- PDF (.pdf)：提取文本内容

#### 1.2 信息提取与结构化

从收集的材料中提取以下结构化信息：

```json
{
  "product_info": {
    "name": "产品名称",
    "type": "平台/系统/工具",
    "domain": "所属领域"
  },
  "requirement_context": {
    "type": "新功能/功能优化/重构/修复",
    "title": "需求标题",
    "description": "需求描述",
    "background": "需求背景",
    "goals": ["目标1", "目标2"]
  },
  "target_users": [
    {
      "role": "角色名称",
      "description": "角色描述",
      "pain_points": ["痛点1", "痛点2"],
      "expectations": ["期望1", "期望2"]
    }
  ],
  "competitive_analysis": [
    {
      "product": "竞品名称",
      "features": ["功能1", "功能2"],
      "reference": "参考内容摘要"
    }
  ]
}
```

#### 1.3 交互式澄清

**关键设计点**：在生成前主动向用户澄清模糊信息

使用`ask_user_question`工具确认以下关键信息：

**1. 功能优先级（MoSCoW分类）**
```
请问此需求的优先级属于哪一类?
选项:
  - A. Must have (必须有,阻塞问题)
  - B. Should have (应该有,重要但不阻塞)
  - C. Could have (可以有,锦上添花)
  - D. Won't have (不会有,明确不做的)
```

**2. 用户角色确认**
```
请问哪些角色会使用此功能? (可多选)
选项:
  - A. 普通用户
  - B. 管理员
  - C. 开发者
  - D. 运维人员
  - E. 其他(请说明)
```

**3. 技术约束**
```
是否有技术栈或架构限制?
选项:
  - A. 无特殊限制
  - B. 必须使用特定技术栈(请说明)
  - C. 需要考虑兼容性(请说明)
```

**4. 非功能需求关注点**
```
本需求特别关注哪些非功能需求? (可多选)
选项:
  - A. 性能
  - B. 安全
  - C. 可用性
  - D. 可扩展性
  - E. 兼容性
```

**5. 竞品策略**
```
对于竞品功能,您的策略是?
选项:
  - A. 借鉴竞品,实现类似功能
  - B. 差异化定位,突出独特价值
  - C. 超越竞品,提供更好的体验
  - D. 不参考竞品
```

将所有澄清问答记录到文件：`prd/{需求标题}_{时间戳}_references/交互记录/{时间戳}_clarifications.md`

### 阶段二：PRD结构化生成

**目标**：基于收集的信息生成PRD各章节内容

#### 2.1 章节生成顺序

采用**自顶向下**的生成策略，按以下顺序生成：

1. **产品概述**
   - 项目背景
   - 业务目标
   - 目标用户
   - 价值主张

2. **功能需求**
   - 功能树构建
   - 功能详细描述
   - 业务流程描述
   - 用户角色与权限

3. **非功能需求**
   - 性能需求
   - 安全需求
   - 可用性需求
   - 兼容性需求

4. **用户体验**
   - 交互流程
   - 界面要求
   - 错误处理

5. **验收标准**
   - 功能验收
   - 性能验收
   - 安全验收

6. **附录**
   - 术语表
   - 参考文档
   - 版本记录

#### 2.2 功能树构建

根据需求描述和竞品分析，构建功能树结构：

```
{需求标题}
├── {功能模块A}
│   ├── {子功能A.1}
│   │   ├── 操作步骤
│   │   └── 业务规则
│   └── {子功能A.2}
├── {功能模块B}
│   ├── {子功能B.1}
│   └── {子功能B.2}
└── {功能模块C}
```

**构建策略**：
- 从需求描述中识别核心功能模块
- 基于竞品分析补充常用功能点
- 考虑完整性和可扩展性
- 使用行业最佳实践

#### 2.3 章节生成与确认

**多轮生成与确认**机制：

每完成一个主要章节，询问用户是否需要调整：

```markdown
# 章节生成确认

已生成"{章节名称}"章节，内容概览：

## 主要内容
{生成的内容摘要}

## 需要确认
{需要用户确认的关键点}

请问是否需要调整?
选项:
  - A. 继续生成下一章节
  - B. 修改本章节
  - C. 补充更多信息
  - D. 跳过本章节
```

记录用户的决策，确保最终PRD一致性。

#### 2.4 AI知识增强策略

**内置行业知识库**：
- SaaS产品通用功能列表
- 常见性能指标参考值
- 安全需求最佳实践
- 用户角色模板

**动态检索**：
- 竞品URL使用`web_fetch`抓取
- 技术方案使用`web_search`搜索最佳实践
- 行业标准查询

**示例增强点**：
1. **性能需求**：根据产品类型提供合理的性能指标
   - SaaS平台：页面加载<2s, API响应<500ms
   - 数据处理：吞吐量TPS、并发用户数
   - 实时系统：延迟<100ms

2. **安全需求**：基于数据敏感度补充安全要求
   - 用户数据：加密存储、传输加密
   - 权限控制：RBAC、最小权限原则
   - 审计日志：操作记录、安全事件追踪

3. **用户体验**：基于用户角色提供UX建议
   - 管理员：配置项集中、批量操作
   - 普通用户：简洁明了、引导式操作
   - 开发者：API文档清晰、调试友好

### 阶段三：PRD完善与优化

**目标**：提升PRD质量和可执行性

#### 3.1 质量检查清单

自动执行以下检查：

- [ ] **功能完整性检查**
  - 核心功能是否完整
  - 辅助功能是否遗漏
  - 配置项、权限控制是否完整
  - 关联功能是否考虑

- [ ] **逻辑一致性检查**
  - 前后描述是否矛盾
  - 业务规则是否冲突
  - 权限设计是否合理
  - 数据流是否清晰

- [ ] **验收标准可测性检查**
  - 每个功能是否有可测试的验收条件
  - 验收标准是否量化、可测量
  - 成功/失败标准是否明确

- [ ] **术语统一性检查**
  - 产品名称、功能名称在全文中一致
  - 专有名词定义是否完整
  - 缩写是否有说明

- [ ] **结构规范性检查**
  - 章节层级是否清晰
  - 标题编号是否连续
  - 是否有明确的目录结构

#### 3.2 补充增强

基于`prd-reviewer`skill的经验，自动补充以下内容：

**1. 测试点建议**
为每个功能添加可测试点：
```markdown
**测试点**：
- [ ] 正常流程：{描述}
- [ ] 异常场景：{描述}
- [ ] 边界条件：{描述}
```

**2. 技术建议**
基于功能描述提供技术实现建议：
```markdown
**技术建议**：
- 建议{方案1}，原因：{理由}
- 考虑{方案2}，优势：{优势}
```

**3. 风险识别**
识别潜在风险点：
```markdown
**潜在风险**：
- 风险1：{描述}，应对措施：{措施}
- 风险2：{描述}，应对措施：{措施}
```

**4. 依赖关系**
标注功能间的依赖：
```markdown
**依赖关系**：
- 前置依赖：{功能A}
- 被依赖：{功能B}
```

## 错误处理

| 异常情况 | 处理方式 |
|----------|----------|
| 初始PRD不存在 | 继续生成，基于需求描述创建 |
| 初始PRD格式不支持 | 提示"不支持的文档格式" |
| 竞品链接无法访问 | 记录到"已知限制"，继续生成 |
| 用户未提供澄清信息 | 使用默认值继续生成 |
| 生成内容为空 | 提示"生成失败，请检查输入" |
| 输出路径无法写入 | 提示"无法写入到指定路径"，询问是否更换 |

## 质量检查标准

PRD生成后，自动评估以下指标：

### 完整性评分 (0-100分)
- 产品概述：15分
- 功能需求：40分
- 非功能需求：20分
- 用户体验：10分
- 验收标准：10分
- 附录：5分

### 可测性评分 (高/中/低)
- 高：所有功能都有明确的验收标准
- 中：主要功能有验收标准，部分缺失
- 低：验收标准不明确或缺失

### 清晰度评分 (高/中/低)
- 高：描述清晰，无歧义
- 中：部分描述需澄清
- 低：描述模糊，存在歧义

## 使用前提

**使用此skill时，用户必须提供以下信息**：

1. **需求类型**：新功能/功能优化/重构/修复
2. **产品名称**：产品名称
3. **需求标题**：需求名称
4. **需求描述**：核心需求说明
5. **输出路径**：PRD输出路径
6. **时间戳**：文件命名时间戳

**可选提供**：
- **初始PRD路径**：已有的简单PRD
- **用户画像**：目标用户群体
- **竞品链接**：竞品产品文档链接
- **参考文档**：相关参考材料

## 注意事项

1. PRD文档输出到 `prd/` 目录下
2. 交互澄清记录保存到 `references/` 目录
3. 生成的PRD建议使用`prd-reviewer`进行评审
4. 对于复杂需求，建议分多次生成，每次聚焦特定模块
5. 生成的PRD仅供参考，需要人工审核和调整
6. 竞品分析需要网络连接，如无法访问会跳过
7. 用户澄清决策会被记录，用于生成过程的可追溯性

## 与其他Skill的协作

- **prd-reviewer**：生成后可调用`prd-reviewer`进行质量评审
- **doc-generator**：功能完成后可使用`doc-generator`生成帮助文档
- **browser-use**：竞品分析时可用于抓取竞品界面截图
- **web_fetch**：用于抓取竞品文档内容
- **web_search**：用于搜索行业最佳实践

## 进阶使用

### 批量生成PRD

如果有多个需求需要生成PRD，可以：
1. 创建一个需求清单文件
2. 批量调用此skill
3. 统一管理和评审

### 定制化模板

如果团队有特定的PRD格式要求：
1. 修改 `templates/prd-template.md`
2. 保持结构一致
3. 调整章节顺序和内容要求

### 知识库扩展

如需扩展行业知识库：
1. 在skill目录下创建 `knowledge/` 目录
2. 添加特定领域的知识文档
3. 在生成逻辑中引用

## 性能优化建议

- 竞品分析可缓存结果，避免重复抓取
- 大型PRD可分章节生成，提高响应速度
- 常用功能点可预生成模板，减少重复计算
- 交互确认可批量进行，减少交互次数
