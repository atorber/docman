# DocMan 部署与启动助手

本 skill 用于帮助用户在本地首次拉取 DocMan 后，一键完成环境检查、依赖安装、服务启动与可用性验证。

## 适用场景

- 用户刚下载 DocMan 到本地，需要完成首次启动
- 启动失败，需要自动排查常见问题（Node 版本、依赖缺失、端口占用等）
- 需要标准化输出“可执行的下一步人工操作指引”

## 输入约定

建议用户在项目根目录调用，并可提供可选参数：

| 参数 | 格式 | 必填 | 说明 |
|------|------|------|------|
| 项目根目录 | `/path/to/aihc-master` | ❌ | 默认使用当前工作目录 |
| 前端端口 | `3000` | ❌ | 默认 3000 |
| 后端端口 | `3001` | ❌ | 默认 3001 |
| 是否重装依赖 | `是/否` | ❌ | 默认否；为“是”时可删除 lock 后重装 |

## 执行目标

1. 检测并确认运行环境是否满足要求  
2. 自动修复可修复问题  
3. 安装并启动前后端服务  
4. 验证服务是否可访问  
5. 无法自动修复时，给出分步骤人工处理建议

## 执行流程

### 阶段一：环境检测

1. 确认当前目录包含 `backend/`、`web/`、`skills/`、`README.md`
2. 检查命令可用性：`node`、`npm`
3. 检查 Node 版本（建议 `>=18`）
4. 检查 npm 版本（建议 `>=8`）

若不满足要求：
- 不要继续安装依赖
- 直接输出阻塞原因与安装建议（例如升级 Node）

### 阶段二：依赖安装与修复

按顺序执行：

1. 在 `backend/` 下执行 `npm install`
2. 在 `web/` 下执行 `npm install`

若安装失败，按以下优先顺序尝试自动修复一次：

1. 网络抖动：重试安装  
2. npm cache 异常：`npm cache verify` 后重试  
3. lock 冲突（仅用户允许重装依赖时）：删除 `node_modules` 与 lock 后重装

若仍失败：
- 停止自动重试
- 输出错误摘要 + 建议人工操作命令

### 阶段三：端口与进程检查

默认检查：
- 前端：`3000`
- 后端：`3001`

处理规则：

1. 若端口空闲：直接启动服务
2. 若端口占用但是 DocMan 旧进程：提示复用（或按用户要求重启）
3. 若端口占用且非 DocMan 进程：给出人工处理建议（换端口或结束占用进程）

### 阶段四：启动服务

在两个终端会话分别启动：

1. `backend/` 执行 `npm run dev`
2. `web/` 执行 `npm run dev`

启动后验证：

- 后端健康检查：`http://localhost:3001/health`
- 前端可访问：`http://localhost:3000`

### 阶段五：结果反馈

成功时输出：

- 前后端访问地址
- 已启动命令与工作目录
- 后续建议（如“打开 /records 检查最近记录”）

失败时输出：

- 失败阶段（环境/依赖/端口/启动）
- 已尝试的自动修复动作
- 必要人工操作步骤（分点、可直接复制执行）

## 人工介入指引模板

当需要人工操作时，按此结构输出：

1. **问题现象**：一句话说明  
2. **可能原因**：1~3 条  
3. **建议操作**：可复制命令（按顺序）  
4. **验证方式**：如何确认问题解决  

示例（端口占用）：

```bash
# 查看 3001 端口占用
lsof -i :3001

# 结束占用进程（替换 PID）
kill -9 <PID>
```

## 注意事项

- 不要在未通过环境检测时盲目执行安装/启动
- 不要无限重试同一失败命令（最多重试 1 次）
- 不要修改业务代码，仅处理部署启动相关问题
- 若涉及系统权限、公司网络代理、证书等问题，需明确提示用户手动介入

## 与其他 Skill 协作

- `doc-consistency-verifier`：服务启动后可执行帮助文档诊断
- `doc-generator`：服务启动后可生成帮助文档 Prompt
- `prd-generator`：服务启动后可生成 PRD Prompt
- `prd-reviewer`：服务启动后可执行 PRD 评审
