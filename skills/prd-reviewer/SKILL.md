# PRD评审工具

本skill用于全面评审产品需求文档(PRD)，从多个专业视角发现需求中的问题、风险和改进点，帮助提升需求质量。

## 适用场景

- 产品需求文档发布前的评审
- 技术评审前的需求检查
- 迭代规划时的需求确认
- 需求变更影响分析

## 调用方式

用户通过prompt提供以下参数：

| 参数 | 格式 | 必填 | 说明 |
|------|------|------|------|
| 原始文档路径 | `prd/xxx/xxx.md` | ✅ | 相对于prd目录的文档路径 |
| 文档名称 | `xxx.md` | ✅ | 文件名（不含路径） |
| 评审视角 | `A, B, C, D, E` | ❌ | 需要评审的视角列表，默认全部 |
| 维度编号 | `A1, A2, ..., E3` | ❌ | 需要评审的维度编号列表，默认全部 |
| 项目背景 | `文字描述` | ❌ | 可选，项目背景信息、约束条件等 |
| 技术栈信息 | `文字描述` | ❌ | 可选，当前系统技术栈信息 |
| 自定义要求 | `文字描述` | ❌ | 可选，额外的评审要求 |
| 时间戳 | `YYYYMMDD_HHMMSS` | ✅ | 输出文件命名的时间戳 |

**prompt示例**：
```
# PRD评审

## 文档信息
- 原始文档路径: prd/用户中心/用户登录优化.md
- 文档名称: 用户登录优化.md

## 评审配置
- 评审视角: A, B, C, D, E
- 维度编号: A1, A2, A3, A4, A5, B1, B2, B3, C1, C2, D1, D2, E1, E2, E3
- 项目背景: 这是一个面向企业用户的SaaS产品，主要用户是中小型企业
- 技术栈信息: 后端使用Java Spring Boot，前端使用React，数据库MySQL
- 自定义要求: 重点评审安全相关需求

## 输出配置
- 时间戳: 20260428_100000
```

**注意**：
- 评审视角使用大写字母：A(产品经理)、B(研发工程师)、C(架构师)、D(测试工程师)、E(文档质量)
- 维度编号格式：视角字母+数字，如 A1、B2、C3
- 如果不提供评审视角和维度编号，则执行全部32个维度的评审
- 时间戳格式必须严格为 `YYYYMMDD_HHMMSS`
- 时间戳参数仅用于文件名，不能用作 timeline 中的 `start_time`

## 评审维度

### A. 产品经理视角（10个维度）

#### A1. 需求完整性
- 业务场景是否全面覆盖
- 功能点是否完整定义
- 是否存在遗漏的功能模块
- 需求范围边界是否清晰

**检查要点**：
- 核心功能流程是否完整
- 辅助功能是否遗漏
- 配置项、权限控制是否完整
- 关联功能是否考虑

#### A2. 业务价值清晰度
- 业务目标是否明确
- 价值指标是否可量化
- 是否与公司战略对齐
- 用户价值是否清晰

**检查要点**：
- 是否有明确的业务目标陈述
- 成功指标是否定义
- 预期收益是否有数据支撑

#### A3. 用户场景覆盖
- 正常流程场景
- 异常流程场景
- 边界条件场景
- 不同用户角色场景

**检查要点**：
- 用户画像是否清晰
- 使用场景描述是否完整
- 是否考虑不同用户群体的差异

#### A4. 边界条件定义
- 输入边界限制
- 数据边界限制
- 状态边界限制
- 时间边界限制

**检查要点**：
- 输入字段长度、格式限制
- 数值范围限制
- 并发限制
- 超时处理

#### A5. 验收标准明确性
- 功能验收标准
- 性能验收标准
- 兼容性验收标准
- 安全验收标准

**检查要点**：
- 每个功能是否有可测试的验收条件
- 验收标准是否量化、可测量
- 成功/失败标准是否明确

#### A6. 优先级合理性
- 功能优先级划分
- 依赖关系识别
- 迭代规划合理性

**检查要点**：
- 优先级是否有依据（MoSCoW原则）
- 是否考虑功能间依赖关系
- 是否符合迭代规划节奏

#### A7. 非功能需求
- 性能需求
- 可用性需求
- 安全需求
- 可扩展性需求

**检查要点**：
- 性能指标是否具体（响应时间、TPS等）
- 可用性目标是否明确（SLA）
- 安全需求是否完整

#### A8. 竞品/对标分析
- 竞品功能对比
- 差异化定位
- 行业最佳实践参考

**检查要点**：
- 是否有竞品分析
- 差异化优势是否清晰
- 是否借鉴行业最佳实践

#### A9. 成本收益分析
- ROI预估
- 资源投入评估
- 运营成本预估

**检查要点**：
- 开发工作量预估
- 运维成本预估
- 收益预估依据

#### A10. 上线风险评估
- 合规风险
- 运营风险
- 技术风险
- 回滚方案

**检查要点**：
- 是否有风险识别
- 风险应对措施是否充分
- 是否有回滚预案

### B. 研发工程师视角（8个维度）

#### B1. 技术可行性
- 现有技术栈支持
- 技术难点识别
- 方案可行性评估

**检查要点**：
- 是否需要引入新技术
- 技术难点是否有解决方案
- 是否有技术验证

#### B2. 实现复杂度评估
- 工作量预估合理性
- 技术债务识别
- 代码复用可能性

**检查要点**：
- 开发工作量是否可预估
- 是否需要重构现有代码
- 是否可以复用现有组件

#### B3. 数据模型设计
- 数据结构合理性
- 字段完整性
- 索引设计
- 数据迁移方案

**检查要点**：
- 数据表设计是否完整
- 字段类型是否合理
- 是否考虑历史数据处理

#### B4. 接口设计完整性
- API设计规范
- 参数定义完整性
- 错误码设计
- 接口文档要求

**检查要点**：
- 接口列表是否完整
- 请求/响应结构是否定义
- 错误处理机制是否完整

#### B5. 性能需求可实现性
- 性能指标可达成性
- 性能瓶颈识别
- 优化方案设计

**检查要点**：
- 性能目标是否可达
- 是否有性能测试方案
- 是否需要缓存、异步等优化

#### B6. 安全需求完整性
- 认证授权机制
- 数据加密要求
- 敏感信息处理
- 接口安全防护

**检查要点**：
- 权限控制是否完整
- 敏感数据是否加密
- 是否有安全审计需求

#### B7. 兼容性要求
- 系统版本兼容性
- 浏览器兼容性
- 设备兼容性
- 接口兼容性

**检查要点**：
- 支持的版本范围
- 向后兼容性要求
- 接口版本管理

#### B8. 可维护性考量
- 代码可读性要求
- 日志埋点需求
- 监控告警需求
- 运维操作需求

**检查要点**：
- 是否有日志规范要求
- 关键指标监控需求
- 运维工具需求

### C. 架构师视角（6个维度）

#### C1. 系统架构影响
- 对现有架构的冲击
- 模块划分合理性
- 系统边界划分

**检查要点**：
- 是否影响现有系统架构
- 新模块如何融入现有架构
- 系统边界是否清晰

#### C2. 扩展性设计
- 未来扩展预留
- 插件化设计
- 配置化能力

**检查要点**：
- 是否考虑未来扩展场景
- 是否有配置化需求
- 是否需要插件机制

#### C3. 技术选型合理性
- 技术方案选择依据
- 引入新组件评估
- 技术栈一致性

**检查要点**：
- 技术选型是否有依据
- 是否与现有技术栈一致
- 新技术引入的风险评估

#### C4. 数据流设计
- 数据流转路径
- 数据一致性保障
- 数据同步机制

**检查要点**：
- 数据流向是否清晰
- 是否有数据一致性要求
- 跨系统数据同步方案

#### C5. 系统依赖分析
- 外部依赖识别
- 依赖稳定性评估
- 降级容错方案

**检查要点**：
- 依赖的外部系统列表
- 依赖失败的影响分析
- 是否有降级方案

#### C6. 技术风险评估
- 技术难点识别
- 不确定性因素
- 备选方案

**检查要点**：
- 技术风险点列表
- 风险应对方案
- 是否有Plan B

### D. 测试工程师视角（5个维度）

#### D1. 测试点可提取性
- 功能测试点清晰度
- 可设计测试用例
- 测试覆盖范围

**检查要点**：
- 每个功能是否有明确测试点
- 是否可以编写测试用例
- 测试范围是否明确

#### D2. 测试数据需求
- 测试数据准备要求
- 数据隔离策略
- 数据构造方案

**检查要点**：
- 是否需要特殊测试数据
- 数据准备工作量
- 数据隔离需求

#### D3. 异常场景覆盖
- 错误场景定义
- 边界值测试场景
- 负面测试场景

**检查要点**：
- 错误场景是否完整
- 边界值是否定义
- 异常处理是否覆盖

#### D4. 回归影响分析
- 影响范围评估
- 回归测试范围
- 自动化测试需求

**检查要点**：
- 改动影响哪些模块
- 回归测试范围
- 是否需要新增自动化用例

#### D5. 兼容性测试需求
- 浏览器测试范围
- 设备测试范围
- 系统版本测试范围

**检查要点**：
- 兼容性测试矩阵
- 测试环境需求
- 特殊兼容性场景

### E. 文档质量视角（3个维度）

#### E1. 格式规范性
- 目录结构
- 标题层级
- 版本记录

**检查要点**：
- 是否有清晰的目录结构
- 标题层级是否规范
- 是否有版本变更记录

#### E2. 术语一致性
- 专有名词统一
- 缩写规范
- 业务术语定义

**检查要点**：
- 术语是否前后一致
- 缩写是否有说明
- 是否有术语表

#### E3. 表述无歧义
- 需求描述清晰
- 无模糊表述
- 无理解歧义

**检查要点**：
- 是否有"适当"、"合理"等模糊词
- 逻辑关系是否清晰
- 是否有多解表述

## 执行流程

### 步骤1：读取并解析文档

1. 根据`原始文档路径`读取文档内容
2. 解析文档结构（标题层级、章节、表格等）
3. 提取关键信息：
   - 需求背景和目标
   - 功能清单和描述
   - 用户流程和场景
   - 验收标准
   - 非功能需求
   - 附录和参考资料

### 步骤2：执行多视角评审

根据`评审视角`和`维度编号`，依次执行各维度的评审：

#### 2.1 产品经理视角评审
- 识别业务目标和使用场景
- 检查功能完整性
- 分析验收标准
- 评估风险和优先级

#### 2.2 研发工程师视角评审
- 评估技术可行性
- 分析实现复杂度
- 检查数据模型和接口设计
- 评估性能和安全需求

#### 2.3 架构师视角评审
- 分析系统架构影响
- 评估扩展性设计
- 审查技术选型
- 识别依赖和风险

#### 2.4 测试工程师视角评审
- 提取测试点
- 分析异常场景
- 评估回归影响
- 确定兼容性测试范围

#### 2.5 文档质量视角评审
- 检查格式规范
- 验证术语一致性
- 识别歧义表述

### 步骤3：生成评审报告

1. 汇总各视角发现的问题
2. 按严重程度（高/中/低）分类
3. 按视角组织评审结论
4. 提供详细的改进建议
5. 给出评审结论（通过/有条件通过/不通过）

### 步骤4：生成过程记录

使用`时间戳`构建输出文件名：`timeline/prd/${文档路径无后缀}_${时间戳}_timeline.json`

## 问题优先级定义

| 优先级 | 说明 | 示例 |
|--------|------|------|
| **高（阻塞）** | 必须解决，否则无法进入开发 | 需求缺失、技术不可行、安全漏洞 |
| **中（重要）** | 建议解决，影响开发效率或质量 | 验收标准不明确、接口设计不完整 |
| **低（一般）** | 可选优化，改善文档质量 | 格式不规范、术语不一致 |

## 使用前提

**使用此skill时，用户必须提供以下信息**：

1. **要评审的PRD文档**：文档的完整路径（相对于prd目录）
2. **时间戳**：用于生成输出文件名

**可选提供**：
- **评审视角和维度**：默认全部32个维度
- **项目背景**：帮助更好地理解需求上下文
- **技术栈信息**：帮助评估技术可行性
- **自定义评审要求**：特殊的评审关注点

如果用户未提供文档路径，询问用户补充后再继续执行。

## 注意事项

1. PRD文档需要放在 `prd/` 目录下
2. 评审报告输出到 `report/prd/` 目录
3. 过程记录输出到 `timeline/prd/` 目录
4. 评审结论仅供参考，最终决策由团队共同确定
5. 对于复杂的PRD，建议分多次评审，每次聚焦特定视角
