# DocMan 文人

一个面向产品文档全生命周期的智能助手，覆盖帮助文档诊断、帮助文档生成、PRD生成与PRD多视角评审，并提供统一的最近记录聚合与详情跳转能力。

## 产品介绍

### 背景与问题

在日常的文档维护过程中，我们经常遇到以下问题：

- **文档过时**：系统UI已经更新，但文档截图和描述还停留在旧版本
- **内容错误**：语法错误、错别字、描述不清晰等问题影响阅读体验
- **流程缺失**：操作步骤不完整，新用户按照文档操作可能失败
- **信息不一致**：文档各章节描述前后矛盾，术语使用不统一
- **维护困难**：大量文档缺乏系统性的质量检查手段，问题发现依赖用户反馈

此外，在需求评审过程中，也常面临以下问题：

- **需求质量参差不齐**：缺乏统一的评审标准和流程
- **视角单一**：评审往往只从产品或开发单方面出发，缺乏多维度审视
- **问题遗漏**：评审维度不全面，导致问题在开发阶段才暴露

### 解决方案

本项目是一个**文档全能助手**，提供四大核心功能：

1. **文档诊断**：通过自动化的方式对产品帮助文档进行全方位质量检查
2. **PRD评审**：从多个专业视角评审产品需求文档，发现需求和风险
3. **帮助文档生成**：基于 PRD 与控制台 URL 生成帮助文档 Prompt
4. **PRD生成**：基于需求信息、竞品链接与本地参考文档生成 PRD Prompt

### 核心能力

#### 一、文档诊断（18维度）
   - 文档与系统一致性检查
   - 语法错误和错别字检测
   - 与常识违背和前后文冲突检查
   - 过时信息和失效链接检测
   - 敏感信息泄露扫描
   - 代码示例错误检查
   - 截图完整性验证
   - 格式一致性检查
   - 文档完整性评估
   - 以及更多...

#### 二、PRD评审（32维度，5大视角）
从产品经理、研发工程师、架构师、测试工程师、文档质量五个专业视角评审PRD：

**产品经理视角（10维度）**
- 需求完整性、业务价值清晰度、用户场景覆盖
- 边界条件定义、验收标准明确性、优先级合理性
- 非功能需求、竞品/对标分析、成本收益分析、上线风险评估

**研发工程师视角（8维度）**
- 技术可行性、实现复杂度评估、数据模型设计
- 接口设计完整性、性能需求可实现性、安全需求完整性
- 兼容性要求、可维护性考量

**架构师视角（6维度）**
- 系统架构影响、扩展性设计、技术选型合理性
- 数据流设计、系统依赖分析、技术风险评估

**测试工程师视角（5维度）**
- 测试点可提取性、测试数据需求、异常场景覆盖
- 回归影响分析、兼容性测试需求

**文档质量视角（3维度）**
- 格式规范性、术语一致性、表述无歧义

#### 三、与实际系统对比验证
   - 支持指定目标系统URL
   - 自动打开浏览器进行界面一致性检查
   - 验证菜单、按钮、文案等是否与实际系统一致
   - 支持复用已登录浏览器，减少重复登录

#### 四、可视化历史管理
   - Web界面直观展示文档目录结构
   - 完整的诊断/评审历史记录
   - 支持查看每次诊断的详细信息（报告、timeline、修复后文档）
   - **支持URL分享跳转**，可通过链接直接打开指定文档/诊断记录

#### 五、诊断报告锚点跳转
   - 各维度诊断详情中的问题数可点击
   - 点击后自动滚动到诊断报告对应章节锚点
   - 快速定位具体问题描述

#### 六、统一最近记录中心
   - "最近记录"统一聚合四类来源：帮助文档诊断、帮助文档生成、PRD生成、PRD评审
   - 支持按来源筛选，并通过来源图标快速区分记录类型
   - "查看详情"按来源跳转到对应页面（诊断/评审可携带参数直达）

#### 七、一键生成Prompt
   - 配置诊断/评审/帮助文档生成/PRD生成参数
   - 自动生成完整的 Prompt
   - 可直接在 Comate 中执行对应任务

### 适用场景

**文档诊断**
- 定期对产品文档进行质量检查
- 新版本上线前检查文档一致性
- 对新编写的文档进行质量把关
- 用户反馈文档问题时快速定位

**PRD评审**
- 产品需求文档发布前的评审
- 技术评审前的需求检查
- 迭代规划时的需求确认
- 需求变更影响分析

## 工作原理

DocMan 采用「**Web 参数配置 + Prompt 生成 + 本地 Agent 执行 Skill**」的模式：

1. 在 DocMan Web 界面中选择任务类型（文档诊断、帮助文档生成、PRD生成、PRD评审）并配置参数  
2. DocMan 后端根据配置生成标准化 Prompt（包含输入路径、输出路径、维度/视角等）  
3. 在本地 Agent（如 **Claude Code**、**OpenClaw** 等）中执行该 Prompt  
4. Agent 调用 DocMan 预置 Skill（`docman-bootstrap`、`doc-consistency-verifier`、`doc-generator`、`prd-generator`、`prd-reviewer`）完成部署/启动与诊断/生成/评审  
5. 执行结果输出到项目目录（如 `report/`、`timeline/`、`new/`），并在 DocMan 的“最近记录”中统一查看和跳转

> 说明：DocMan 本身不直接执行大模型任务，而是负责参数编排、Prompt 生成与结果可视化管理。

## 使用流程

1. **按目录准备文档**
   - 帮助文档诊断输入文档放在 `raw/`
   - PRD评审 / PRD生成 / 本地参考文档放在 `prd/`
   - 目录结构需保持与项目约定一致，否则无法正确发现文件和生成记录

2. **打开 Web 界面**
   访问 http://localhost:3000

3. **选择任务与输入**
   - 文档诊断：在左侧文档树选择待诊断文档
   - PRD评审：进入 PRD评审页选择待评审 PRD
   - 帮助文档生成 / PRD生成：在对应页面配置输入参数、输出位置、可选高级项

4. **生成 Prompt**
   在对应页面点击生成，复制生成的 Prompt

5. **在本地 Agent 执行**
   在 Claude Code、OpenClaw 等本地 Agent 中执行 Prompt，触发对应 Skill 完成任务

6. **回到 DocMan 查看结果**
   执行完成后刷新页面，通过“最近记录”按来源筛选并查看详情

### 文档诊断典型流程（示例）

1. **选择文档**
   在左侧文档目录树中选择需要诊断的文档

2. **查看诊断历史**
   "诊断历史"面板显示该文档的历史诊断记录

3. **生成诊断 Prompt**
   点击"生成诊断"TAB，配置诊断参数后生成Prompt

4. **执行诊断**
   复制生成的 Prompt，在本地 Agent 中执行诊断

5. **查看结果**
   任务完成后刷新页面，或从"最近记录"按来源筛选并查看详情

## 快速开始

### 环境要求

- Node.js >= 18.0
- npm >= 8.0

### 安装依赖

```bash
# 安装后端依赖
cd backend
npm install

# 安装前端依赖
cd ../web
npm install
```

### 启动服务

```bash
# 启动后端服务 (端口 3001)
cd backend
npm run dev

# 启动前端服务 (端口 3000)
cd web
npm run dev
```

### 一键启动（推荐）

在项目根目录执行：

```bash
# 一键启动前后端
npm run dev

# 一键启动（后端调试模式，默认 inspect 9229）
npm run dev:debug
```

### 使用部署启动 Skill（推荐）

首次在本地部署时，建议在项目根目录通过本地 Agent 调用 `docman-bootstrap`：

- 自动检测运行环境（Node/npm、目录结构）
- 自动安装前后端依赖
- 自动启动前后端服务并校验可用性
- 无法自动修复时给出分步骤人工操作建议

可直接复制的一句话 Prompt：`请使用当前目录的 docman-bootstrap skill 启动当前 DocMan 项目。`

## 路由说明

项目使用 React Router 实现多个主导航页面：

| 路由 | 说明 |
|------|------|
| `/` 或 `/diagnose` | 文档诊断首页（左侧文档树 + 右侧内容区） |
| `/records` | 最近记录页面（统一聚合多来源记录） |
| `/docgen` | 帮助文档生成页面 |
| `/?nav=prdgen` | PRD生成页面 |
| `/prd-review` | PRD评审首页（左侧PRD文档树 + 右侧内容区） |

**注意**：各主页面为独立路由，切换时不携带跨页参数。

- 文档诊断页：`/?doc=路径&record=timeline路径&tab=preview`
- 最近记录页：`/records`
- 帮助文档生成页：`/docgen`
- PRD生成页：`/?nav=prdgen`
- PRD评审页：`/prd-review?doc=路径&record=timeline路径`

### URL 分享跳转

- 选择文档或诊断记录后，URL自动更新
- 复制当前URL即可分享给他人
- 打开链接可直达对应文档和诊断结果
- 示例格式：
  - 打开指定文档：`/?doc=操作指南/工作流/创建工作流.md`
  - 打开指定诊断记录：`/?doc=操作指南/工作流/创建工作流.md&record=timeline/操作指南/工作流/创建工作流_20260427_214131_timeline.json`

## 诊断标准说明

### 高优先级问题
- 敏感信息泄露（密码、密钥、Token等）
- 失效链接/图片
- 与系统不一致
- 代码示例错误
- 缺少必要的警告/提示

### 中优先级问题
- 语法错误
- 错别字
- 前后文冲突
- 过时信息
- 缺失步骤
- 版本兼容性问题

### 低优先级问题
- 格式不一致
- 描述不清晰
- 截图缺失
- 冗余内容
- 命名不规范
- 文档完整性问题

## 注意事项

### 文档诊断
1. 原始文档需要放在 `raw/` 目录下
2. 诊断过程中可能需要登录目标系统，请提前准备好账号
3. 支持复用已登录浏览器进行验证，减少重复登录操作
4. 敏感信息泄露检测是强制高优先级，任何密码、密钥等敏感词都会高亮提醒
5. 建议定期进行文档诊断，保持文档质量

### PRD评审
1. PRD文档需要放在 `prd/` 目录下
2. 评审报告输出到 `report/prd/` 目录
3. 过程记录输出到 `timeline/prd/` 目录
4. 评审结论仅供参考，最终决策由团队共同确定
5. 对于复杂的PRD，建议分多次评审，每次聚焦特定视角

### 帮助文档生成与PRD生成
1. 生成类功能通过 Prompt 驱动执行，页面负责参数配置与 Prompt 生成
2. PRD生成支持配置输出路径、填写多个竞品链接、并多选本地参考文档（`prd/` 下 `.md`）
3. 最近记录可按来源筛选并跳转到对应页面，便于统一追踪全链路任务
