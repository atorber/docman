# DocMan

一个面向产品帮助文档的智能诊断系统，帮助发现文档中的各种质量问题。同时支持产品需求文档(PRD)的多视角评审。

## 产品介绍

### 背景与问题

在日常的文档维护过程中，我们经常遇到以下问题：

- **文档过时**：系统UI已经更新，但文档截图和描述还停留在旧版本
- **内容错误**：语法错误、错别字、描述不清晰等问题影响阅读体验
- **流程缺失**：操作步骤不完整，新用户按照文档操作可能失败
- **信息不一致**：文档各章节描述前后矛盾，术语使用不统一
- **维护困难**：大量文档缺乏系统性的质量检查手段，问题发现依赖用户反馈

此外，在需求评审过程中，也常面临以下问题：

- **需求质量参差不齐**：缺乏统一的评审标准和流程
- **视角单一**：评审往往只从产品或开发单方面出发，缺乏多维度审视
- **问题遗漏**：评审维度不全面，导致问题在开发阶段才暴露

### 解决方案

本项目是一个**文档全能助手**，提供两大核心功能：

1. **文档诊断**：通过自动化的方式对产品帮助文档进行全方位质量检查
2. **PRD评审**：从多个专业视角评审产品需求文档，发现需求和风险

### 核心能力

#### 一、文档诊断（18维度）
   - 文档与系统一致性检查
   - 语法错误和错别字检测
   - 与常识违背和前后文冲突检查
   - 过时信息和失效链接检测
   - 敏感信息泄露扫描
   - 代码示例错误检查
   - 截图完整性验证
   - 格式一致性检查
   - 文档完整性评估
   - 以及更多...

#### 二、PRD评审（32维度，5大视角）
从产品经理、研发工程师、架构师、测试工程师、文档质量五个专业视角评审PRD：

**产品经理视角（10维度）**
- 需求完整性、业务价值清晰度、用户场景覆盖
- 边界条件定义、验收标准明确性、优先级合理性
- 非功能需求、竞品/对标分析、成本收益分析、上线风险评估

**研发工程师视角（8维度）**
- 技术可行性、实现复杂度评估、数据模型设计
- 接口设计完整性、性能需求可实现性、安全需求完整性
- 兼容性要求、可维护性考量

**架构师视角（6维度）**
- 系统架构影响、扩展性设计、技术选型合理性
- 数据流设计、系统依赖分析、技术风险评估

**测试工程师视角（5维度）**
- 测试点可提取性、测试数据需求、异常场景覆盖
- 回归影响分析、兼容性测试需求

**文档质量视角（3维度）**
- 格式规范性、术语一致性、表述无歧义

#### 三、与实际系统对比验证
   - 支持指定目标系统URL
   - 自动打开浏览器进行界面一致性检查
   - 验证菜单、按钮、文案等是否与实际系统一致
   - 支持复用已登录浏览器，减少重复登录

#### 四、可视化历史管理
   - Web界面直观展示文档目录结构
   - 完整的诊断/评审历史记录
   - 支持查看每次诊断的详细信息（报告、timeline、修复后文档）
   - **支持URL分享跳转**，可通过链接直接打开指定文档/诊断记录

#### 五、诊断报告锚点跳转
   - 各维度诊断详情中的问题数可点击
   - 点击后自动滚动到诊断报告对应章节锚点
   - 快速定位具体问题描述

#### 六、最近记录直达报告
   - "最近记录"列表中的"查看详情"可直接跳转到文档诊断页
   - 自动打开该文档最新一次诊断报告
   - 无需手动切换TAB和选择历史记录

#### 七、一键生成Prompt
   - 配置诊断/评审参数
   - 自动生成完整的Prompt
   - 可直接在Comate中执行诊断/评审

### 适用场景

**文档诊断**
- 定期对产品文档进行质量检查
- 新版本上线前检查文档一致性
- 对新编写的文档进行质量把关
- 用户反馈文档问题时快速定位

**PRD评审**
- 产品需求文档发布前的评审
- 技术评审前的需求检查
- 迭代规划时的需求确认
- 需求变更影响分析

## 快速开始

### 环境要求

- Node.js >= 18.0
- npm >= 8.0

### 安装依赖

```bash
# 安装后端依赖
cd backend
npm install

# 安装前端依赖
cd ../web
npm install
```

### 启动服务

```bash
# 启动后端服务 (端口 3001)
cd backend
npm run dev

# 启动前端服务 (端口 3000)
cd web
npm run dev
```

### 使用流程

1. **打开Web界面**
   访问 http://localhost:3000

2. **选择文档**
   在左侧文档目录树中选择需要诊断的文档

3. **查看诊断历史**
   "诊断历史"面板显示该文档的历史诊断记录

4. **生成诊断Prompt**
   点击"生成诊断"TAB，配置诊断参数后生成Prompt

5. **执行诊断**
   复制生成的Prompt，在Comate中执行诊断

6. **查看结果**
   诊断完成后刷新页面，或从"最近记录"直接查看诊断报告

## 路由说明

项目使用 React Router 实现多个独立主TAB：

| 路由 | 说明 |
|------|------|
| `/` 或 `/diagnose` | 文档诊断首页（左侧文档树 + 右侧内容区） |
| `/records` | 最近记录页面（全站诊断历史列表） |
| `/prd-review` | PRD评审首页（左侧PRD文档树 + 右侧内容区） |
| `/prd-records` | PRD评审记录页面 |

**注意**：各主TAB为独立路由，切换时不携带跨页参数。

- 文档诊断页：`/?doc=路径&record=timeline路径&tab=preview`
- 最近记录页：`/records`
- PRD评审页：`/prd-review?doc=路径&record=timeline路径`

### URL 分享跳转

- 选择文档或诊断记录后，URL自动更新
- 复制当前URL即可分享给他人
- 打开链接可直达对应文档和诊断结果
- 示例格式：
  - 打开指定文档：`/?doc=操作指南/工作流/创建工作流.md`
  - 打开指定诊断记录：`/?doc=操作指南/工作流/创建工作流.md&record=timeline/操作指南/工作流/创建工作流_20260427_214131_timeline.json`

## 诊断标准说明

### 高优先级问题
- 敏感信息泄露（密码、密钥、Token等）
- 失效链接/图片
- 与系统不一致
- 代码示例错误
- 缺少必要的警告/提示

### 中优先级问题
- 语法错误
- 错别字
- 前后文冲突
- 过时信息
- 缺失步骤
- 版本兼容性问题

### 低优先级问题
- 格式不一致
- 描述不清晰
- 截图缺失
- 冗余内容
- 命名不规范
- 文档完整性问题

## 输出文件说明

| 目录 | 内容 | 命名格式 |
|------|------|----------|
| raw/ | 原始文档 | - |
| new/ | 修复后的文档 | [文档名]_[时间戳]_new.md |
| report/ | 诊断报告（含锚点） | [文档名]_[时间戳]_report.md |
| timeline/ | 诊断过程记录 | [文档名]_[时间戳]_timeline.json |
| screenshots/ | 诊断过程中截取的系统截图 | 按文档路径组织 |

## 注意事项

### 文档诊断
1. 原始文档需要放在 `raw/` 目录下
2. 诊断过程中可能需要登录目标系统，请提前准备好账号
3. 支持复用已登录浏览器进行验证，减少重复登录操作
4. 敏感信息泄露检测是强制高优先级，任何密码、密钥等敏感词都会高亮提醒
5. 建议定期进行文档诊断，保持文档质量

### PRD评审
1. PRD文档需要放在 `prd/` 目录下
2. 评审报告输出到 `report/prd/` 目录
3. 过程记录输出到 `timeline/prd/` 目录
4. 评审结论仅供参考，最终决策由团队共同确定
5. 对于复杂的PRD，建议分多次评审，每次聚焦特定视角
